MX8 provides several CLI tools for deploying and monitoring distributed data pipelines:

**Core runtime tools:**
- **mx8-coordinator**: Control plane server that manages leases, dataset ownership, and job coordination
- **mx8d-agent**: Worker process that runs on each node to execute data pipeline tasks
- **mx8-tui**: Terminal UI for monitoring live job state and diagnosing issues

**Dataset utilities:**
- **mx8-pack-s3**: Packs many small S3 objects into MX8 tar shards with canonical manifest
- **mx8-snapshot-resolve**: Resolves dataset links to pinned snapshots and prints manifest hash
- **mx8-seed-s3**: Uploads local files to S3 (used for testing and seeding data)

## Architecture

A typical MX8 deployment consists of:

1. **One coordinator process** acting as the job control plane
2. **One agent process per node** handling local memory caps and feeding work to runtimes
3. **Optional TUI** for real-time monitoring and diagnostics

The coordinator manages leases and dataset ownership but never proxies dataset data bytes. Agents and runtimes fetch data directly from storage (S3 or filesystem).

## Quick Start

### Build the binaries

```bash
cargo build -p mx8-coordinator --features s3
cargo build -p mx8d-agent --features s3
cargo build -p mx8-tui
```

Omit `--features s3` if using filesystem-only datasets.

### Start the coordinator

```bash
MX8_COORD_BIND_ADDR=0.0.0.0:50051 \
MX8_WORLD_SIZE=8 \
MX8_DATASET_LINK='s3://your-bucket/dataset/@refresh' \
MX8_MANIFEST_STORE_ROOT='s3://your-bucket/manifests/' \
target/release/mx8-coordinator
```

### Start agents on each node

```bash
MX8_COORD_URL='http://coordinator-host:50051' \
MX8_JOB_ID='prod-job-001' \
MX8_NODE_ID='node-0' \
MX8_BATCH_SIZE_SAMPLES=512 \
MX8_MAX_INFLIGHT_BYTES=134217728 \
MX8_MAX_PROCESS_RSS_BYTES=68719476736 \
target/release/mx8d-agent
```

### Monitor with TUI

```bash
cargo run -p mx8-tui -- \
  --coord-url http://coordinator-host:50051 \
  --job-id prod-job-001
```

## Configuration Pattern

All CLI tools follow a consistent configuration pattern:

- Every CLI argument has a corresponding environment variable
- Environment variables use the `MX8_` prefix
- CLI flags override environment variables
- Timing values are in milliseconds
- Default values are production-safe for small deployments

## Network Topology

The coordinator binds a gRPC port (default `0.0.0.0:50051`) that every agent connects to. Each agent:

- Runs on one node
- Talks to the coordinator for leases, heartbeats, and progress reporting
- Fetches data directly from the manifest store (S3 or filesystem)

Place the coordinator on a node with stable network paths to all agents and the manifest store. The default heartbeat interval is 1 second, so network reliability matters more than raw latency.

## Failure Handling

When an agent dies or stops heartbeating:

1. Coordinator waits for lease TTL to expire (default 10 seconds)
2. Unfinished range `[cursor, end)` is automatically requeued
3. Remaining agents pick up the work and continue draining
4. No manual intervention needed unless the agent doesn't return

When the coordinator dies with HA enabled:

1. Leader transition continues control-plane progress
2. Agents reconnect automatically
3. Heartbeats and lease issuance resume

## Production Baseline

Before running production jobs, set these explicitly:

- `MX8_MAX_PROCESS_RSS_BYTES`: Set as org-wide policy, not per-job ad-hoc values
- `MX8_WORLD_SIZE`: Use explicit value rather than relying on defaults
- `MX8_DATASET_LINK`: Use `@refresh` or pinned `@sha256:` hash for production
- `MX8_MANIFEST_STORE_ROOT`: Back by S3 or durable disk, not local ephemeral storage
- `MX8_GRPC_MAX_MESSAGE_BYTES`: Set to same value on coordinator and all agents

Run the readiness suite before new deployments:

```bash
./scripts/prod_readiness.sh
```

## Dataset Utility Tools

MX8 provides additional CLI utilities for dataset management and snapshot resolution.

### mx8-pack-s3

Packs many small S3 objects into MX8 tar shards and writes a canonical manifest. Run this once on a raw dataset prefix to produce a packed dataset that MX8 can load efficiently without large S3 LIST operations.

```bash
cargo run -p mx8-snapshot --features s3 --bin mx8-pack-s3 -- \
  --pack-in 's3://my-bucket/raw/train/' \
  --pack-out 's3://my-bucket/mx8/train/'
```

**Key arguments:**
- `--pack-in` / `MX8_PACK_IN`: S3 input prefix (required)
- `--pack-out` / `MX8_PACK_OUT`: S3 output prefix (required)
- `--shard-mb` / `MX8_PACK_SHARD_MB`: Target shard size in MiB (default: 512)
- `--label-mode` / `MX8_S3_LABEL_MODE`: `auto`, `none`, or `imagefolder` (default: auto)

On success, prints: `samples=<n> shards=<n> manifest_key=<key> manifest_hash=<sha256>`

### mx8-snapshot-resolve

Resolves a dataset link to a pinned snapshot and prints the manifest hash. Useful for pre-pinning a snapshot before a job starts.

```bash
cargo run -p mx8-snapshot --features s3 --bin mx8-snapshot-resolve -- \
  --dataset-link 's3://my-bucket/mx8/train/@refresh' \
  --manifest-store-root '~/.mx8/manifests'
```

**Key arguments:**
- `--dataset-link` / `MX8_DATASET_LINK`: Plain path, `@refresh`, or `@sha256:<hash>` (required)
- `--manifest-store-root` / `MX8_MANIFEST_STORE_ROOT`: Filesystem path or S3 prefix (default: `~/.mx8/manifests`)

On success, prints: `manifest_hash: <sha256>`

### mx8-seed-s3

Uploads a local file as a single S3 object. Used by deterministic local gates to seed test data into MinIO before running a gate.

```bash
cargo run -p mx8-runtime --features s3 --bin mx8-seed-s3 -- \
  --bucket mx8-demo \
  --key data.bin \
  --file ./data.bin
```

**Key arguments:**
- `--bucket` / `MX8_MINIO_BUCKET`: Destination bucket (default: mx8-demo)
- `--key` / `MX8_MINIO_KEY`: Destination object key (default: data.bin)
- `--file` / `MX8_SEED_FILE`: Local file path to upload (required)

<Note>
These utilities are typically used during dataset preparation, not during runtime. For Python-based packing, use `mx8.pack()` or `mx8.pack_dir()` instead.
</Note>
